Comment la page a été « WebMCP-ifiée »

Le code et la balise ajoutés à la page d'accueil de la Présidence de l'Assemblée nationale pour l'exposer à un agent via navigator.modelContext — sans toucher au contenu visible.

Ce document explique, pas à pas, le code ajouté à la page d'accueil de la Présidence de l'Assemblée nationale pour l'exposer à un agent via WebMCP (le brouillon W3C qui standardise navigator.modelContext). On s'appuie sur la librairie @webmcp-auto-ui/core.

L'idée directrice : ne rien changer au contenu visible. On ajoute une seule balise <script>, et tout le travail se fait en JavaScript en lisant le DOM déjà présent.

1. La seule balise ajoutée à la page

Tout en bas du <body>, juste avant </body>, on insère un unique bloc de script :

    <!-- ... tout le contenu original de la page, inchangé ... -->

    <script>
      /* le code WebMCP, détaillé ci-dessous */
    </script>
  </body>
</html>

C'est la seule modification du balisage. Pas de nouvelle <div>, pas de CSS, rien dans le contenu. On place le script en fin de <body> pour garantir que tout le DOM (agenda, actualités, pied de page) existe déjà quand le code s'exécute : il pourra donc le lire sans attendre.

En production, on écrirait plutôt <script type="module" src="webmcp.js"> ; ici tout est regroupé en ligne pour livrer un seul fichier autonome.

2. Importer les briques de la librairie

import {
  initializeWebMCPPolyfill, // installe navigator.modelContext si absent
  hasNativeWebMCP,          // dit si le navigateur a déjà l'API native
  createToolGroup,          // regroupe nos outils (enregistrement/retrait en bloc)
  jsonResult,               // emballe une valeur JS au format de réponse WebMCP
  listenForAgentCalls,      // pont vers une extension d'agent (postMessage)
} from '@webmcp-auto-ui/core';

Chaque import correspond à une responsabilité précise. On n'utilise rien d'autre.

3. Être ré-exécutable sans tout casser

Si on relance le script (rechargement partiel, ré-injection), on retire d'abord les outils déjà posés :

if (window.__presidenceWebMCP?.group) {
  window.__presidenceWebMCP.group.abort(); // désenregistre tous nos outils
  window.__presidenceWebMCP.stop?.();      // coupe le pont d'écoute
}

window.__presidenceWebMCP est le « hook » qu'on expose à la fin (voir §8) ; sa présence signale qu'un enregistrement précédent existe.

4. Activer l'API WebMCP

initializeWebMCPPolyfill({ allowInsecureContext: true });

Deux cas :

  • Navigateur récent avec WebMCP natif (ex. Chrome 146 + flag) : la fonction détecte l'API et ne fait rien — on utilise le navigator.modelContext du navigateur.
  • Navigateur sans l'API : la librairie installe un polyfill, c'est-à-dire une implémentation JavaScript de navigator.modelContext conforme au brouillon.

allowInsecureContext: true autorise le fonctionnement hors HTTPS (utile en file:// ou http://localhost).

5. Lire le contenu de la page (le « scraping »)

C'est le cœur pédagogique : les outils ne contiennent pas de données en dur, ils lisent le DOM rendu. Trois fonctions extraient l'agenda, les actualités et le contact. Exemple simplifié pour l'agenda :

function extraireAgenda() {
  // 1. on récupère le texte affiché, ligne par ligne
  const lignes = document.body.innerText
    .split('\n').map((l) => l.trim()).filter(Boolean);

  const entrees = [];
  let dateCourante = null;

  for (const ligne of lignes) {
    // 2. un en-tête de jour (« Mercredi 29 avril ») arme une nouvelle date
    const m = ligne.match(JOUR_RE);
    if (m && ligne.length <= 30) { dateCourante = iso(m[2], m[3]); continue; }

    // 3. sinon, sous une date active, la ligne devient une entrée d'agenda
    if (dateCourante) entrees.push({ date: dateCourante, libelle: ligne });
  }
  return entrees;
}

La vraie version est plus robuste (filtres anti-bruit, fusion des lignes de précision comme « Ambassadeur de France… », déduplication des slides du carrousel), mais le principe tient en trois temps : découper le texte → repérer les dates → rattacher chaque ligne à sa date.

Pour les actualités, on s'appuie sur la structure HTML réelle plutôt que sur le texte : les liens du type /activites/<type>/<id> donnent l'identifiant et l'URL, et on retrouve le titre dans la carte qui entoure le lien.

for (const a of document.querySelectorAll('a[href*="/activites/"]')) {
  const m = a.href.match(/\/activites\/([a-z_]+)\/(\d+)/);
  if (!m) continue;
  const id = Number(m[2]);          // ex. 49098
  const type = m[1];                // ex. activite_diplomatique
  // … on lit le titre et la date dans le texte du lien / de sa carte
}

Ces fonctions sont appelées une fois au chargement, et leurs résultats stockés :

const AGENDA = extraireAgenda();
const ACTIVITES = extraireActivites();
const CONTACT = extraireContact();

6. Décrire et enregistrer les outils WebMCP

On crée d'abord un groupe (pour pouvoir tout retirer d'un coup, cf. §3) et une petite fonction qui relaie l'enregistrement vers l'API :

const group = createToolGroup('presidence-an');
const registerFn = (tool, options) =>
  navigator.modelContext.registerTool(tool, options);

Un outil WebMCP est un simple objet : un nom, une description (lue par l'agent pour savoir quand l'utiliser), un schéma d'entrée JSON (les paramètres acceptés) et une fonction execute qui renvoie le résultat.

const OUTILS = [
  {
    name: 'get_agenda_presidente',
    description: "Agenda public de la Présidente, filtrable entre deux dates (AAAA-MM-JJ).",
    inputSchema: {
      type: 'object',
      properties: {
        date_debut: { type: 'string', description: 'AAAA-MM-JJ' },
        date_fin:   { type: 'string', description: 'AAAA-MM-JJ' },
      },
    },
    execute: async ({ date_debut, date_fin } = {}) => {
      const d1 = date_debut || '0000-01-01';
      const d2 = date_fin || date_debut || '9999-12-31';
      // jsonResult emballe la valeur au format attendu par WebMCP :
      // { content: [{ type: 'text', text: '<json>' }] }
      return jsonResult(AGENDA.filter((e) => e.date >= d1 && e.date <= d2));
    },
  },
  // … 6 autres outils : search_agenda, list_activites, get_activite,
  //   get_presidente, list_presidents, get_contact_presse
];

Puis on les enregistre tous :

OUTILS.forEach((t) => group.register(t, registerFn));

À partir d'ici, un agent qui interroge navigator.modelContext voit les 7 outils, lit leur description et peut les appeler avec des arguments validés contre le inputSchema.

Les 7 outils exposés :

Outil Rôle
get_agenda_presidente agenda filtré entre deux dates
search_agenda recherche plein texte dans l'agenda
list_activites actualités, filtrables par type
get_activite détail d'une actualité par identifiant
get_presidente fiche institutionnelle de la Présidente
list_presidents présidents depuis 1958
get_contact_presse coordonnées de la Présidence

7. Pont vers une extension d'agent

const stop = listenForAgentCalls((name, args) => executeToolInternal(name, args));

Certaines extensions (inspecteurs WebMCP) ne lisent pas l'objet JS directement : elles communiquent par window.postMessage. Cette ligne installe l'écouteur qui reçoit « appelle l'outil X avec les arguments Y » et renvoie le résultat. stop permet de couper ce pont (cf. §3).

8. Un hook pratique, autonome

Enfin, on expose un objet global pour tester ou piloter les outils sans dépendre du canal natif :

const EXECUTEURS = new Map(OUTILS.map((t) => [t.name, t.execute]));

async function appeler(name, args) {
  const ex = EXECUTEURS.get(name);
  if (!ex) throw new Error(`Outil inconnu : ${name}`);
  return ex(args || {});            // exécute directement la fonction de l'outil
}

window.__presidenceWebMCP = {
  group, stop,
  natif: hasNativeWebMCP(),
  outils: OUTILS.map((t) => t.name),
  donnees: { agenda: AGENDA, activites: ACTIVITES, contact: CONTACT },
  invoke: (name, args) => appeler(name, args),                         // résultat WebMCP brut
  call:   (name, args) => appeler(name, args)
                            .then((r) => JSON.parse(r.content[0].text)), // données JSON
};

Pourquoi une carte d'exécuteurs locale ? Quand le WebMCP natif est actif, registerTool écrit dans le registre du navigateur, pas dans celui de la librairie. Une fonction d'exécution interne à la librairie ne « verrait » donc pas les outils. En gardant nos propres execute dans EXECUTEURS, le hook fonctionne dans tous les cas (natif comme polyfill).

Exemples d'utilisation depuis la console ou un agent :

await __presidenceWebMCP.call('search_agenda', { query: 'Villeroy' });
await __presidenceWebMCP.call('get_agenda_presidente',
                              { date_debut: '2026-06-01', date_fin: '2026-06-30' });
await __presidenceWebMCP.invoke('get_contact_presse'); // résultat WebMCP complet
